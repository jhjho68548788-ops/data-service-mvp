# 데이터서비스 기획서 (간단히 실행 가능한 버전 / MVP)

## 0. 문서 목적
- 원본 제안서(`데이터서비스기획서.md`)의 아키텍처 축(하이브리드 데이터 커넥터/정규화, 디자인 시스템 테마, AI 주도 기능 조립, 개인화, 권한·보안)을 유지하면서
- 구현에 바로 착수할 수 있도록 **MVP 범위**, **모듈-요구사항 매핑**, **입출력(데이터 계약)**, **최소 실행 절차**를 정리합니다.
- 본 문서는 “AI를 완전 구현”하는 문서가 아니라, **규칙 기반(또는 목업)으로도 end-to-end 동작**하는 “간단 실행 가능한 프로토타입”을 목표로 합니다.

---

## 1. MVP 목표 (End-to-End 데모 기준)
1. 설정 파일로 `데이터 소스`(공공/구독형)를 등록한다.
2. 요청 시 해당 소스를 호출(실제 호출 대신 기본 제공 mock 가능)하고, `정규화 로직`을 통해 표준 스키마로 변환한다.
3. 설정된 `테마(컬러/폰트/컴포넌트 규칙)`를 UI에 적용한다.
4. `기능 빌더`가 요청된 “대시보드/분석 화면” 사양을 받아, 최소 구성(예: 제목, 지표, 차트/테이블 슬롯)을 조립한다.
5. `권한 체크`를 통과한 사용자만 유료 소스/기능을 사용 가능하게 한다.
6. “비용/쿼터/호출빈도”는 MVP에서는 **정책 엔진 + 목업 카운터**로 대체하여, 동작 여부를 검증한다.

---

## 2. 전체 구조 맵 (원본 대비 MVP 단위)
### 2.1 모듈(책임)
- `Data Source Registry`
  - 공공/구독 소스의 API 가이드(또는 mock 핸들러)와 메타를 관리
- `Hybrid Data Connector` (Adapter)
  - 소스 유형별 호출/수집 정책(공공은 오픈키, 구독은 키/쿼터/비용 정책)을 실행
- `Normalization Engine`
  - 입력 데이터를 서비스 표준 스키마로 변환
- `Theme & Style Store`
  - 조직별 테마/컬러 템플릿을 저장하고 기능 빌더/화면 렌더러에 전달
- `Feature Module Repository`
  - 조립 가능한 컴포넌트/블록(지표 카드, 시계열 차트, 상관분석 카드 등)의 명세와 최소 로직을 제공
- `Identity & Permission Store`
  - 사용자 등급/소스 접근/기능 권한을 관리
- `Feature Builder`
  - FeatureSpec을 받아 컴포넌트 블록을 선택/배치(= MVP에서는 규칙 기반으로 시작)
- `Personalization (MVP)`
  - 급변 감지 등 고급 로직은 MVP에서는 생략 가능하되, “추천 버튼/제안 결과”를 목업으로 제공

### 2.2 데이터 저장(최소)
- MVP에서는 DB는 `선택` 사항이지만, 최소 구현을 위해 아래를 **단일 스토어**로 두는 것을 권장
  - `config store` (JSON) + `runtime store` (간단 DB 또는 in-memory)

## 3. 요구사항 목록 및 대응 모듈
| 요구사항(요약) | MVP에서의 처리 방식 | 대응 모듈 |
|---|---|---|
| 공공/구독형 소스 혼합 수집 | mock connector + 정책 스위치 | `Data Source Registry`, `Hybrid Data Connector` |
| 라이선스/쿼터/유효기간 관리 | 호출 시 “정책 통과/차단”을 목업 카운터로 검증 | `Hybrid Data Connector`, `Identity & Permission Store` |
| 어떤 출처든 표준 규격으로 정규화 | 표준 스키마(`CanonicalDataset`)로 변환 어댑터 구현 | `Normalization Engine` |
| 디자인 시스템 테마 적용 | 테마 JSON을 UI 컴포넌트에 반영 | `Theme & Style Store` |
| AI 주도 기능 조립(초기: 규칙 기반) | FeatureSpec -> 컴포넌트 조립(규칙/템플릿) | `Feature Builder`, `Feature Module Repository` |
| 권한/보안 제어 | 사용자별로 접근 가능한 소스/기능 필터링 | `Identity & Permission Store` |
| 개인화/제안 | MVP에서는 “추천 결과”를 목업으로 표시 | `Personalization (MVP)` |
| 무중단 확장(구조 기반) | 새 소스/새 블록은 설정/모듈 추가로 확장 | 전체 모듈 경계 |

## 4. 데이터 계약(필수 스키마)
아래 계약을 지키면 구현 언어/프레임워크가 달라도 동작하는 프로토타입을 만들 수 있습니다.

### 4.1 SourceConfig (데이터 소스 등록)
```json
{

  "type": "public",
  "displayName": "한국은행 ECOS",
  "connector": {
    "kind": "mock",
    "seed": "ecos-demo"
  },
  "ratePolicy": {
    "maxCallsPerHour": 120
  },
  "normalizationProfileId": "macro-index-v1"
}
```

```json
{
  "sourceId": "vendor-steel",
```

### 4.3 FeatureSpec (기능 빌드 요청 사양)

## 5. 간단 실행 아키텍처(권장 Reference)
본 MVP는 “AI”를 완전 대체하지 않고, 먼저 **조립/정규화/권한/테마 적용**을 end-to-end로 검증합니다.

### 5.1 실행 흐름(요청 -> 응답)
1. `POST /build-feature` 로 `FeatureSpec` 제출
2. `Identity`에서 사용자 권한으로 `requiredSources` 허용 여부 확인
3. 허용된 소스만 수집(공공/구독 커넥터)
4. 각 소스를 `Normalization Engine`으로 `CanonicalDataset` 생성
5. `Feature Builder`가 `analysisType` + `layout.cards`에 맞춰 컴포넌트 블록 조립
6. 결과로 `FeatureRuntime`(화면 렌더링 모델) 반환
7. UI는 `themeId`에 따라 토큰을 적용하여 화면 렌더링

### 5.2 필수 엔드포인트(스택 무관 계약)
- `GET /sources` : 등록된 소스 목록 반환
- `GET /themes/{orgId}` : 테마 토큰 반환
- `POST /build-feature`
  - 입력: `FeatureSpec`
  - 출력: `FeatureRuntime`(화면 렌더 모델 + 필요한 데이터셋 참조)

### 5.3 FeatureRuntime(최소 출력 형태)
```json
{
  "runtimeId": "rt-001",
  "themeId": "acme-default",
  "title": "기준금리 vs 철강 가격 상관 대시보드",
  "components": [
    { "type": "kpi-card", "props": { "metric": "latest" } },
    { "type": "timeseries-chart", "props": { "datasetRef": "ds-123", "x": "x", "y": "y" } },
    { "type": "correlation-card", "props": { "datasetRef": "ds-123" } }
  ],
  "datasets": [
    { "datasetId": "ds-123", "canonicalType": "time_series" }
  ]
}
```

## 6. 실행 절차(“간단 실행” 단계)
아래는 코드가 아직 없는 상태에서, 구현자가 바로 따라 만들 수 있도록 “필수 산출물” 중심으로 작성했습니다.

### 6.1 준비물(입력 파일)
1. `sources.json`
2. `themes/{orgId}.json`
3. `permissions.json` (사용자별 허용 소스/기능)
4. `feature-specs/` (데모용 FeatureSpec 몇 개)

### 6.2 데모용 FeatureSpec 최소 세트(권장)
- `feature-corr-ecos-steel-v1` (공공+구독 동시)
- `feature-public-only-v1` (공공만 사용)
- `feature-vendor-blocked-v1` (권한 없는 유료 소스 접근 차단 확인)

### 6.3 검증 기준(데모 체크리스트)
- 동일한 FeatureSpec 요청 시, 권한이 없는 사용자는 `vendor` 소스가 차단된다.
- 소스 타입이 달라도 `CanonicalDataset` 스키마가 동일하게 유지된다.
- 테마 토큰이 화면 컴포넌트(카드/차트)에 반영된다.
- 기능 조립이 항상 동일한 레이아웃 규칙을 따른다(규칙 기반이므로 재현 가능).
